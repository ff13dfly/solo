# 履约生命周期建模 (Fulfillment Lifecycle Modeling)

本文档基于 `import/state-01` 和 `import/state-02` 的状态机设计稿整理，旨在通过抽象的履约状态机指导 Solo·AI 未来微服务的开发与建模。

## 1. 核心理念：状态驱动开发
履约过程被抽象为一个单向推进的生命周期，每个阶段由特定的**推进条件 (Propulsion Conditions)** 触发，并受到 **拦截条件 (Interception Conditions)** 的约束。

在 Solo 架构中，微服务应围绕这些状态节点进行建模，确保：
- **H (Human)**: 精确的审批与审核节点。
- **M (Money)**: 财务归集与逻辑分块。
- **P (Product)**: 货品、库存与实物状态。
- **ERP**: 与底层业务系统的单据同步。
- **AI**: 在全链路上提供风险预警、自动填充与决策辅助。

---

## 2. 履约阶段详解

### 第一阶段：订单创建与订金确认 (Order Creation & Deposit)
*   **状态节点**：创建新订单 -> 待付订金 -> 销售确认。
*   **关键动作**：
    - **H**: 销售主管审核、财务确认账务。
    - **M**: 财务单据（订金确认）。
    - **P**: 销售库存查看（初步不判断严格库存）。
    - **ERP**: 生成预收款单/下销货单。
*   **AI 增强点**：客户画像评分（状态因子）、收款方式风险预警、大环境政策预警。

### 第二阶段：备货与排产阶段 (Preparation & Sourcing)
*   **状态节点**：配货中心接收 -> 订单整合 -> 细部处理（包装等）。
*   **关键动作**：
    - **H**: 内勤主导、供应商确认产能/排期。
    - **P**: 细部处理、包裹打包。
    - **ERP**: 生成采购订单。
*   **推进条件**：订金全额到账 (0%~N%) 或 客户信用额度通过。
*   **AI 增强点**：自动分拣不同工厂/仓库的货物、发货地址智能筛选、产能/排期冲突预测。

### 第三阶段：仓储调配与发货 (Shipping & Dispatch)
*   **状态节点**：从工厂/中心仓出库 -> 将货物中转/装车 -> 发货给客户。
*   **关键动作**：
    - **H**: 仓管调度配货、运货安排。
    - **P**: 货品发出最后状态确认。
    - **ERP**: 生成出库单。
*   **AI 增强点**：出货异常通知、物流节点状态同步。

### 第四阶段：完成与售后服务 (Completion & After-sales)
*   **状态节点**：已收货 -> 售后状态。
*   **关键动作**：
    - **H**: 售后回访、货品取回处理。
    - **M**: 处理退货后的财务结算差额。
    - **ERP**: 生成财务销货单。
*   **AI 增强点**：异常反馈自动归集、满意度调查预分析。

## 3. 系统互操作性建议

### 拦截控制 (Intercepts)
每个状态推进前应有拦截层，例如：
- 财务未确认时，禁止进入备货阶段。
- 供应商产能爆满时，订单自动进入“排队”状态拦截。

### AI 介入规则 (AI Co-pilot)
AI 不应直接修改状态机，而应作为**辅助决策器**：
1. **风险控制**：在收款节点检测 FX (汇率) 风险或客户违约风险。
2. **效率提升**：基于历史数据自动推荐分仓策略。
3. **一致性检查**：在 ERP 同步前校验数据完整性。

## 4. 后续建模路线图
1. **微服务拆分**：考虑是否需要独立的 `fulfillment` 服务负责状态推进，或由 `order` 与 `commodity` 协同。
2. **事件总线**：每个状态跳转应发布 `state_transitioned` 事件，供 AI 模型监听并生成预警。
3. **ERP 映射**：建立严密的单据状态映射表，防止状态孤岛。
